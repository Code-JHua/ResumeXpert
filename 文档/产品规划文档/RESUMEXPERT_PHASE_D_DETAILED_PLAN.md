# ResumeXpert Phase D 细化文档

## 1. 文档定位
本文件用于承接 Phase C 已完成的内容双引擎与模板渲染底座，细化 Phase D“输出与分享中心”的具体目标、范围、任务拆解、接口设计与验收标准。

Phase D 的核心目标不是简单再加一个下载按钮，而是把 ResumeXpert 从“编辑器 + PDF 导出”升级为“统一输出中心”，让同一份简历资产可以面向不同渠道稳定输出。

---

## 2. Phase D 目标

### 产品目标
把 ResumeXpert 从“主要依赖前端下载 PDF”的简历工具，升级为“支持 PDF、Markdown、分享页三种主要输出方式”的职业表达平台。

### 阶段目标
本阶段优先完成：

1. Markdown 导出正式可用
2. 分享页创建、关闭与状态管理
3. 导出日志可查询、可展示
4. 分享访问统计可追踪

本阶段不做：

- DOCX 导出
- 自定义公开链接域名
- 分享页密码保护与高级权限矩阵
- SEO 精细化优化
- 模板市场级别的外部嵌入分享组件

### 完成后的用户价值
- 用户可以围绕同一份简历选择不同输出渠道
- 用户可以把 Markdown 版本交给技术招聘场景使用
- 用户可以生成在线分享链接，而不必每次都重新发 PDF
- 用户可以知道自己导出了什么、什么时候导出、分享页是否被访问

---

## 3. 当前基础与缺口

### 已具备基础
结合当前文档与代码，Phase D 已经有了一部分可复用基础：

- 前端已存在 PDF 导出服务：
  - `frontend/src/services/resumeExportService.js`
- 前端仍支持现有预览渲染链路，可直接复用统一渲染树
- 后端已存在 Markdown 导出接口：
  - `GET /api/resume/:id/export/markdown`
- 后端已存在导出日志模型：
  - `backend/models/exportLogModel.js`
- 后端已存在导出日志接口：
  - `POST /api/resume/:id/exports/log`
  - `GET /api/resume/:id/exports`
- 前端已预留分享管理入口页面：
  - `/share`
  - `frontend/src/pages/ShareManagementPage.jsx`
- Phase C 已完成 Markdown 文档资产化与 `freeBlocks` 承接

### 当前缺口
- PDF 导出仍主要依赖前端本地下载，缺少统一“输出中心”页面
- Markdown 导出虽然已有接口，但还缺正式入口、状态提示和下载体验
- 导出日志虽有模型和接口，但尚未形成用户可见的管理界面
- 分享页模型、公开访问路由、开关控制、访问统计尚未落地
- 还没有“导出行为”和“分享行为”的统一产品心智

### 技术约束
- 当前 PDF 方案基于 `html2pdf.js`，质量与浏览器环境强相关
- 当前后端尚无 `shared_resume_pages` 模型
- 当前分享页未建立匿名访问边界，需要单独设计公开读取接口
- 必须兼容现有简历编辑器、模板预览、Markdown 文档体系，不能破坏现有导入和编辑链路

## 4. Phase D 范围定义

## 4.1 In Scope

- 导出文件云端归档

## 5. Phase D 核心用户流

## 5.1 流程一：PDF 导出
1. 用户进入简历编辑页或输出中心
2. 点击“导出 PDF”
3. 系统调用统一导出服务
4. 浏览器生成并下载 PDF
5. 系统记录一条 `export_logs`
6. 用户可在输出中心看到导出历史

## 5.2 流程二：Markdown 导出
1. 用户进入输出中心
2. 点击“导出 Markdown”
3. 系统请求 `GET /api/resume/:id/export/markdown`
4. 若 Markdown 文档已存在：
   - 返回内容
   - 浏览器下载 `.md` 文件或复制到剪贴板
5. 若 Markdown 文档未就绪：
   - 明确提示当前状态
   - 引导用户先进入 Markdown 页面同步
6. 系统记录一条 `export_logs`

## 5.3 流程三：开启分享页
1. 用户进入输出中心或分享管理页
2. 点击“创建分享链接”
3. 系统为简历创建或更新一条分享记录
4. 系统返回公开访问链接
5. 用户可复制链接、查看状态、手动关闭分享
6. 每次公开访问会累计访问统计

## 5.4 流程四：关闭分享页
1. 用户在分享管理页关闭分享
2. 系统把分享状态改为 `disabled` 或 `archived`
3. 原链接访问后显示“分享已关闭”
4. 历史统计仍保留供用户查看

## 6. 数据与状态设计

## 6.1 `export_logs` 使用规则
当前项目已存在 `ExportLog` 模型，Phase D 继续沿用并扩展其产品含义。

建议核心字段：

- `userId`
- `resumeId`
- `format`
- `templateId`
- `resumeVersionId`
- `status`
- `metadata`

`format` 在本阶段主要包括：

- `pdf`
- `markdown`
- `share`

`status` 延续现有设计：

- `pending`
- `success`
- `failed`

建议 `metadata` 在 Phase D 进一步承载：

- `fileName`
- `triggerSource`
- `sharePageId`
- `errorMessage`
- `exportDurationMs`

## 6.2 建议新增 `shared_resume_pages`
为 Phase D 建议新增实体：

- `shared_resume_pages`

建议字段：

- `userId`
- `resumeId`
- `resumeVersionId`
- `slug`
- `title`
- `isEnabled`
- `visibility`
- `lastPublishedAt`
- `lastViewedAt`
- `viewCount`
- `uniqueVisitorCount`
- `lastSnapshot`
- `themeSnapshot`

字段含义建议：

- `slug`: 分享页公开访问标识
- `isEnabled`: 当前是否允许访问
- `visibility`: 预留后续扩展，首期可仅支持 `public`
- `lastSnapshot`: 发布时的内容快照，避免公开页被编辑中的临时状态污染
- `themeSnapshot`: 发布时模板和主题快照

## 6.3 分享状态定义
建议状态逻辑采用轻量模式：

- `draft`
- `published`
- `disabled`

状态含义：

- `draft`: 已创建分享配置，但尚未正式公开
- `published`: 分享页已公开可访问
- `disabled`: 链接已停用，但记录仍保留

如果不想新增状态字段，也可以先用：

- `isEnabled`
- `lastPublishedAt`

来表达最小可用状态机。

## 6.4 分享访问统计结构
首期不建议一开始就建立重型埋点系统。

建议采用两层结构：

1. `shared_resume_pages` 上聚合字段
2. 可选新增 `share_visit_logs` 记录明细

若首期只做最小可用版本，可先仅保留聚合字段：

- `viewCount`
- `lastViewedAt`
- `uniqueVisitorCount`

唯一访客统计可先采用弱准确方案：

- 基于匿名 cookie 或访问指纹去重
- 不追求分析平台级绝对精确

## 7. Phase D 详细开发任务表

## Epic D1：统一输出中心落地

### D1-1 新增输出中心入口
- 目标：让导出和分享不再散落在各页面按钮中
- 前端实现建议：
  - 在简历编辑页增加“输出 / 分享”入口
  - 或将 `/share` 扩展为完整输出中心页
  - 页面至少包括：
    - PDF 导出卡片
    - Markdown 导出卡片
    - 分享管理卡片
    - 导出历史列表
- 验收：
  - 用户可从简历编辑页稳定进入输出中心
  - 不同输出方式有统一操作区

### D1-2 统一输出动作来源标记
- 目标：让日志知道导出是从哪里触发的
- 实现建议：
  - 在 `recordExportLog` 中增加 `triggerSource`
  - 可选值：
    - `editor`
    - `output_center`
    - `share_management`
- 验收：
  - 导出日志能区分触发入口

### D1-3 输出中心列表能力
- 目标：让用户看到最近导出与分享活动
- 前端/后端实现：
  - 使用现有 `GET /api/resume/:id/exports`
  - 页面展示：
    - 格式
    - 状态
    - 模板
    - 时间
    - 错误信息
- 验收：
  - 用户可查看最近导出历史
  - 失败记录可被明确识别

---

## Epic D2：Markdown 导出正式化

### D2-1 Markdown 导出入口接入
- 目标：把已有导出接口转成真实可用功能
- 前端实现：
  - 输出中心增加“导出 Markdown”按钮
  - 支持：
    - 下载 `.md`
    - 复制 Markdown 内容
- 验收：
  - 已存在 Markdown 文档时，用户可成功下载 Markdown 文件

### D2-2 Markdown 未就绪状态处理
- 目标：避免导出失败时用户不知道下一步该做什么
- 后端/前端实现：
  - 沿用当前 `status = not_ready`
  - 前端根据响应显示引导：
    - 前往 `/resume/:id/markdown`
    - 执行“从表单同步 Markdown”
- 验收：
  - 没有 Markdown 文档时不会出现空下载
  - 用户能被明确引导到补齐动作

### D2-3 Markdown 导出日志补齐
- 目标：让 Markdown 导出与 PDF 导出一样进入统一日志
- 实现建议：
  - 成功导出时记录 `format = markdown`
  - 未就绪和异常都记录失败日志
- 验收：
  - Markdown 导出行为可在历史列表中查看

---

## Epic D3：PDF 导出体验标准化

### D3-1 统一 PDF 导出触发点
- 目标：减少“每个页面各自导出”的分散体验
- 前端实现：
  - 复用 `resumeExportService.js`
  - 把主要导出动作收敛到输出中心和编辑器头部
- 验收：
  - PDF 导出入口清晰，行为一致

### D3-2 PDF 导出日志标准化
- 目标：把现有 PDF 导出日志写得更完整
- 实现建议：
  - 补充 `fileName`
  - 补充 `templateId`
  - 记录失败原因与导出耗时
- 验收：
  - 成功/失败日志信息完整

### D3-3 PDF 导出异常提示优化
- 目标：降低 `html2pdf.js` 失败时的用户困惑
- 前端实现：
  - 明确提示：
    - 浏览器渲染失败
    - 页面元素未准备完成
    - 模板内容过长或样式异常
- 验收：
  - 失败时页面有可理解提示，不只是控制台报错

---

## Epic D4：分享页主链路

### D4-1 新增分享页模型与接口
- 目标：建立分享功能的后端主资产
- 后端建议新增：
  - 模型：`shared_resume_pages`
  - 接口：
    - `POST /api/resume/:id/share`
    - `GET /api/resume/:id/share`
    - `PUT /api/resume/:id/share`
    - `POST /api/resume/:id/share/toggle`
- 行为：
  - 创建分享配置
  - 生成 `slug`
  - 保存发布快照
- 验收：
  - 每份简历可创建或更新一个分享配置

### D4-2 新增公开分享访问路由
- 目标：支持匿名用户访问公开简历页
- 前端/后端建议：
  - 前端路由：
    - `/s/:slug`
  - 后端公开接口：
    - `GET /api/public/share/:slug`
- 页面要求：
  - 只读展示
  - 不暴露编辑操作
  - 样式复用现有模板渲染能力
- 验收：
  - 未登录用户可访问公开链接
  - 关闭分享后公开链接不可继续查看正文

### D4-3 分享页快照机制
- 目标：避免用户编辑中的中间状态实时暴露到公开页
- 实现建议：
  - 发布时保存一份内容快照与模板快照
  - 公开页优先渲染快照
  - 编辑后是否重新发布由用户手动控制
- 验收：
  - 分享页内容稳定
  - 用户知道“当前编辑内容”和“当前公开内容”可能不同

### D4-4 分享状态管理页
- 目标：把现有 `/share` 占位页升级为正式页面
- 页面建议展示：
  - 分享状态
  - 公开链接
  - 最后发布时间
  - 累计访问数
  - 开启 / 关闭按钮
  - 重新发布按钮
- 验收：
  - 用户可在一个页面完成分享开关和链接管理

---

## Epic D5：分享访问统计

### D5-1 公开访问计数
- 目标：记录分享页被查看的基本次数
- 后端实现：
  - 公开访问成功时增加：
    - `viewCount`
    - `lastViewedAt`
- 验收：
  - 打开公开页后计数增长

### D5-2 唯一访客统计
- 目标：提供比总浏览量更有价值的基础指标
- 实现建议：
  - 首期采用 cookie 或简易访客指纹
  - 聚合回写到 `uniqueVisitorCount`
- 验收：
  - 同一访客重复访问不会无限放大唯一访问数

### D5-3 分享访问概览展示
- 目标：让用户能看到最基础的分享效果反馈
- 前端展示建议：
  - 总浏览量
  - 唯一访客数
  - 最后访问时间
- 验收：
  - 用户能在分享管理页查看访问概览

## 11. 风险与应对

### 风险一：前端 PDF 导出稳定性受浏览器影响
应对：
- 保留当前方案作为首期实现
- 通过日志记录失败原因
- 在后续阶段再评估服务端 PDF 渲染

### 风险二：公开分享暴露编辑中的未完成内容
应对：
- 使用“发布快照”机制
- 区分“当前编辑版”和“当前公开版”

### 风险三：分享统计实现过重
应对：
- 首期只做聚合字段
- 先满足“用户能看到有没有被访问”

### 风险四：Markdown 导出与表单当前状态不一致
应对：
- 导出前提示 Markdown 同步状态
- 当状态为 `outdated` 时明确提醒用户先同步

## 12. Phase D 完成定义
当满足以下条件时，可视为 Phase D 完成：

- 用户可从统一入口导出 PDF
- 用户可正式导出 Markdown
- 每次导出都会留下可查询日志
- 用户可创建、复制、关闭分享链接
- 匿名用户可访问公开分享页
- 用户可查看分享访问统计
- 输出与分享能力不破坏现有编辑、导入、Markdown 双编辑链路

---

## 13. 推荐开发顺序

### 第一批
1. `D2-1 Markdown 导出入口接入`
2. `D2-2 Markdown 未就绪状态处理`
3. `D3-2 PDF 导出日志标准化`
4. `D1-3 输出中心列表能力`

原因：
这批任务可以最大化复用当前已有代码，最快把“输出中心”做成真实可见能力。
