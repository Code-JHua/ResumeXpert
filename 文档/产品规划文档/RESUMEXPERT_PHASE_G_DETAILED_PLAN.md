# ResumeXpert Phase G 细化文档

## 1. 文档定位
本文件用于承接 Phase F 已完成的“模板平台化”，细化 Phase G“高自由度创作”的目标、范围、任务拆解、接口设计与验收标准。

Phase G 的核心目标不是再增加几个模板皮肤，而是让模板从“可选”升级为“可创作”：用户可以组合区块、控制布局、形成自定义模板资产，并把模板提交到社区审核流中；同时补齐 Word 导入导出，使创作结果可以进入更多办公场景。

---

## 2. Phase G 目标

### 产品目标
把 ResumeXpert 的模板能力从“模板平台”进一步推进为“模板创作工作台”，开始具备块级排版、轻量布局创作和社区发布审核的雏形。

### 阶段目标
本阶段优先完成：

1. 模板片段系统
2. 自定义模板编辑器
3. 布局模式
4. 社区发布和审核流
5. Word 导入导出 MVP

本阶段不做：

- 像 Figma 一样的自由拖拽画布
- 模板源码在线编辑器
- 沙箱执行第三方模板代码
- Word 样式 100% 还原导入
- 多管理员权限系统

### 完成后的用户价值
- 用户可以真正“编辑模板结构”，而不是只能换主题色
- 模板可以控制区块顺序、显隐、标题和主栏/侧栏归属
- 用户可以创建适合自己行业和经历结构的模板副本
- 社区模板开始具备“提交 - 审核 - 发布”的产品链路
- 简历可导入 DOCX、导出 DOCX，适配更多办公协作场景

## 3. 当前基础与缺口

### 已具备基础
- Template 资产模型已存在
- 官方模板中心已存在
- 模板收藏、复制、套用能力已存在
- 主题参数协议已存在
- 模板社区预留字段已存在
- Markdown / PDF 导入与多格式输出中心已存在

### 当前缺口
- 模板仍然缺少真正的“区块级结构配置”
- 用户还不能编辑模板布局模式
- 社区字段虽然预留了，但没有提交和审核流
- 模板仍主要依赖固定渲染器，缺少灵活布局渲染器
- Word 导入导出还没有真正接入

### 技术约束
- 当前系统仍以前端本地渲染器为核心，不能直接动态下发任意模板代码
- 简历数据模型还是结构化字段优先，因此自由布局必须基于这些结构化区块来做
- 社区审核首期没有完整角色权限体系，需要采用轻量工作台方案

## 5. Phase G 核心用户流

## 5.1 流程一：创建高自由度模板
1. 用户进入 `/templates`
2. 选择 `Flexible Studio` 或基于任意模板复制
3. 在模板片段系统中调整：
   - 区块标题
   - 区块顺序
   - 区块显隐
   - 主栏 / 侧栏归属
4. 切换单栏 / 双栏布局
5. 同步调整主题参数
6. 保存模板资产

## 5.2 流程二：将自定义模板应用到简历
1. 用户在模板工作台选择一份简历
2. 点击“套用到当前简历”
3. 系统将：
   - `rendererKey`
   - `templateId`
   - `theme settings`
   - `layoutMode`
   - `blockConfig`
   写入 `Resume.template.settings`
4. 编辑器和输出中心复用同一套模板配置回放

## 5.3 流程三：提交社区审核
1. 用户在模板工作台编辑完模板
2. 填写提交说明
3. 点击“提交社区审核”
4. 系统将模板状态切换为社区待审核
5. 审核台中出现该模板

## 5.4 流程四：审核社区模板
1. 运营/当前登录用户进入模板中心的审核台
2. 查看待审核模板
3. 填写审核备注
4. 选择“通过”或“驳回”
5. 审核通过后，该模板进入公开社区模板集合

## 5.5 流程五：Word 导入与导出
1. 用户在导入中心上传 `.docx`
2. 系统提取原始文本，映射成待确认简历草稿
3. 用户确认后生成 Resume
4. 用户在输出中心点击“导出 DOCX”
5. 系统生成结构化 Word 文档供外部使用

## 6. 数据与状态设计

## 6.1 `Template.blockSchema`
建议新增：

- `layoutMode`
- `availableLayouts`
- `blocks`

其中 `blocks` 结构建议包括：

- `key`
- `title`
- `visible`
- `column`
- `order`

### 设计意图
- 让模板拥有“区块配置层”
- 支持固定模板和自由模板共存
- 用户修改布局时，不需要修改渲染器源码

## 6.2 `Resume.template.settings`
Phase G 建议新增/约定：

- `layoutMode`
- `blockConfig`

### 设计意图
- 模板套用后，简历自身能带着模板创作结果走完整条链路
- 编辑器预览、输出中心、公开分享页都能一致回放

## 6.3 社区审核状态
`Template.communityMeta` 中建议使用：

- `reviewStatus`
  - `reserved`
  - `pending`
  - `approved`
  - `rejected`
- `submitterNote`
- `submittedAt`
- `reviewerName`
- `reviewedAt`
- `reviewNotes`

### 设计意图
- 把“社区发布”从占位字段升级为实际流程字段

## 6.4 Word 导入导出资产规则
导入时：
- `ResumeImport.sourceType = word`

导出时：
- 输出格式 `docx`
- 导出日志记为 `format = docx`

## 7. Phase G 详细开发任务表

## Epic G1：模板片段系统

### G1-1 新增模板区块配置结构
- 目标：为模板增加可编辑的区块层
- 后端实现：
  - `Template.blockSchema`
- 前端实现：
  - `templateBlocks` 默认配置
- 验收：
  - 模板可配置区块顺序、显隐和归属

### G1-2 统一区块配置回放
- 目标：让简历应用模板后能携带区块配置
- 后端实现：
  - `POST /api/templates/:id/apply` 支持 `blockConfig`
- 前端实现：
  - 套用模板时写入布局配置
- 验收：
  - 模板套用后，区块配置可在预览中生效

## Epic G2：自定义模板编辑器

### G2-1 新增灵活布局渲染器
- 目标：提供一个能吃区块配置和布局模式的通用渲染器
- 前端实现：
  - 新增 `FlexibleTemplate`
  - 注册 `rendererKey = flex`
- 验收：
  - 自定义模板可基于 `flex` 渲染器稳定渲染

### G2-2 新增模板片段编辑器
- 目标：让用户直接编辑模板区块
- 前端实现：
  - 新增 `TemplateBlockConfigurator`
- 验收：
  - 用户可在 UI 中调整区块标题、顺序、显隐和栏位

### G2-3 模板中心升级为创作工作台
- 目标：把模板中心升级为创作入口
- 前端实现：
  - 模板配置、区块配置、主题配置、预览联动
- 验收：
  - 用户可在一个页面完成模板创作和套用

## Epic G3：布局模式

### G3-1 支持单栏 / 双栏布局
- 目标：让自由模板不再受固定布局限制
- 实现：
  - `layoutMode = single | two-column`
  - 渲染器根据模式切换布局
- 验收：
  - 同一模板可切换单栏和双栏布局

### G3-2 区块主栏 / 侧栏归属
- 目标：让布局模式不仅是切换页面骨架，还能控制区块位置
- 实现：
  - `column = main | sidebar | full`
- 验收：
  - 用户可把技能、教育等区块切换到侧栏

## Epic G4：社区发布和审核

### G4-1 模板提交社区审核
- 目标：让个人模板进入社区流
- 后端实现：
  - `POST /api/templates/:id/submit-community`
- 前端实现：
  - 提交说明输入框
  - 提交按钮
- 验收：
  - 模板可进入待审核状态

### G4-2 社区审核队列
- 目标：把待审核模板集中展示
- 后端实现：
  - `GET /api/templates/review-queue`
- 前端实现：
  - 审核台列表
- 验收：
  - 可查看社区模板审核队列

### G4-3 审核通过 / 驳回
- 目标：形成最小闭环
- 后端实现：
  - `POST /api/templates/:id/review`
- 前端实现：
  - 通过 / 驳回按钮
  - 审核备注
- 验收：
  - 模板可切换到 approved / rejected

## Epic G5：Word 导入导出

### G5-1 DOCX 导入
- 目标：补齐常见办公文档入口
- 后端实现：
  - `POST /api/imports/docx`
  - 使用 `.docx` 文本抽取服务
- 前端实现：
  - 导入中心新增 Word 模式
- 验收：
  - 用户可上传 Word 文档并进入确认流程

### G5-2 DOCX 导出
- 目标：让简历可输出为标准 Word 文档
- 后端实现：
  - `GET /api/resume/:id/export/docx`
- 前端实现：
  - 输出中心新增 DOCX 导出按钮
- 验收：
  - 用户可下载 `.docx` 文件

## 11. 风险与应对

### 风险一：自由布局过早走向像素级拖拽，复杂度失控
应对：
- Phase G 明确采用“块级配置 + 布局模式”，不做画布式拖拽

### 风险二：社区审核缺少角色系统
应对：
- 首期采用轻量审核台
- 后续若进入多人运营，再补角色权限

### 风险三：Word 导入精度不稳定
应对：
- Word 导入首期走“文本抽取 + 人工确认”
- 不承诺复杂表格和样式还原

### 风险四：自定义模板导致输出链路不一致
应对：
- 编辑器、输出中心、公开页统一复用 `RenderResume`
- `Resume.template.settings` 作为统一回放来源

## 12. Phase G 完成定义
当满足以下条件时，可视为 Phase G 完成：

- 模板已支持区块配置
- 已存在灵活布局渲染器
- 模板中心可完成自定义模板创作
- 模板支持单栏 / 双栏布局切换
- 模板可提交社区审核并在审核台处理
- 导入中心已支持 DOCX 导入
- 输出中心已支持 DOCX 导出

## 13. 推荐开发顺序

### 第一批
1. `G1-1 新增模板区块配置结构`
2. `G2-1 新增灵活布局渲染器`
3. `G3-1 支持单栏 / 双栏布局`

原因：
先把“高自由度创作”的最小骨架搭起来。

### 第二批
1. `G2-2 新增模板片段编辑器`
2. `G2-3 模板中心升级为创作工作台`
3. `G1-2 统一区块配置回放`

原因：
这批任务让创作能力真正可用。

### 第三批
1. `G4-1 模板提交社区审核`
2. `G4-2 社区审核队列`
3. `G4-3 审核通过 / 驳回`
4. `G5-1 DOCX 导入`
5. `G5-2 DOCX 导出`

原因：
这批任务负责把模板创作从“个人可玩”推进到“可流转、可输出”。

## 14. 下一步建议
Phase G 完成后，最适合进入下一阶段的两条线是：

1. 真正的拖拽式模板画布
2. 多角色社区模板市场

如果继续推进，最合适的下一步是：

`先把 Flexible Studio 继续细化为真正的布局画布，再补社区模板的权限和审核后台。`
